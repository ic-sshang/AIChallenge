File Mapping
AMS files have a version attached to it. From the code we should be able to support version 1.14, 1.16 versions. 1.10, 1.11 and 1.12 are considered obsolete versions. No information is provided for other versions, we will need to review.
We receive a daily FULL Delta file which contains all the data for accounts for the day. This file is processed by “AMSFileSplit” which generates  type R invoice, adjustment and customer update files. Deciding if the Line is a CUF/Adjustment vs a type R invoice is based on the Due date (AccountRecord8)  and a field called statement date (AccountRecord55). Due date is required and the statement date should be empty for the record to be considered a type R record. 
Add additional check before file split processing. IC can add either Advanced Setting or CRM option to enable this functionality on Billers. When we receive a full delta file, we check to see if customer record already exists only on accounts with no statement dates. If it's true, skip creating a dummy invoice record. Otherwise, create customer record using data from full delta file, then create a dummy invoice record using today's date as invoice date and due date. InvoiceNumber should be something like "AccountPrefix-AccountNumber-DueDate", BalanceDue and TotalAmountDue should be $0.
AccountPrefix = Initial, AccountNumber - same as data mapping based on delta file, DueDate = Today's date
Note: No changes on current file split process, only add statement date check, customer record and invoice creation before determine record goes into adjustment.

For BID 2272 service version (01.14) to only use the value of the AccountRecord35 to determine if an account should receive payment blocks. They will no longer rely on AccountRecord32 and Advanced Setting 63.
