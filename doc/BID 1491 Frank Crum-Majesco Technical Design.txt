User Interaction and Design
Data Flow -Invoice
XML invoice file will be sent via SFTP to Invoice Cloud which will be converted into Invoice Cloud BIF.
Data Flow - Down Payments
Down Payments will be processed through Cloud Pay v2. Quote payments will have the invoice number format: “INV-<QuoteNO>-<PayplanID>-<YYMMDDHH24MISS>”
Data Flow - Adjustments
Balance Updates will be received through Real Time Data Refresh
Data Flow - Payment
Payments will be sent via Data Pump
Data Flow - Void/ACH Reject
Voids will be handled through IC. Any voids will be data pumped back to Majesco.
ACH Rejects will be data pumped back to Majesco. 
Data Flow - Credit
Refunds and Credits will be processed by Majesco by using “Process Credit Card Credit with PaymentGUID” and “Process ACH Credit with PaymentGUID” webservice calls.
Data Flow - Change Log
Paperless and Autopay flags will be sent via Data Pump to Majesco. Emails will not be provided to Majesco.
Data Flow - Bill Presentment
Bill presentment will be done realtime with PDF Linking.
Payer Experience
Payers will access Invoice Cloud from the Biller website (URL redirect). Payers will use the IC Virtual Site and Customer Portal.
Biller Experience
Agents will process downpayments through Majesco with Cloud Pay v2. This will allow them to sign insured up for Autopay, Paperless, and Pay by Text as well. Biller Portal will be used for reporting and payment Voids.

System Jira
 
TBD
Currently API authentication is being done with Basic Authentication with provided Username and Password
API Authentication to be switched from Basic Authentication to JWT Token Based Authentication using the Bearer Token
Currently, when we access our API using basic authentication, session data is created for each API call. This adds significant overhead to our security module. In contrast, token-based authentication allows the API to remain stateless, eliminating the need to store session data for each user. This approach is particularly advantageous in distributed or cloud-based systems, where scalability is critical.
The overhead associated with basic authentication has caused multiple instances of higher latency in the security module, negatively affecting API performance. For example, the Production issue on 10/21 was directly attributed to this problem. Increased security latency led to a ripple effect on our billing API, where requests began queuing, impacting overall processing.
