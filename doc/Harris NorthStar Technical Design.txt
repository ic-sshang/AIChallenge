Add new flag named “ccflag“ to block credit card in RTDR:
BlockPaymentsCC
: if “ccflag“ exists(for different billers) and is equal to “Y“, we block customer credit card. Else we unblock the customer if the “ccflag“ is not “Y“ and is not an empty value. This flag is not used by all billers. If it’s not present, no action on CC payment.
BlockPaymentsEFT
: We block the customer if the “nsfflag” is equal to “y”. Else, we unblock the customer if the “nsfflag” is not “y” and is not an empty value.
BlockPaymentsEFT and BlockPaymentsCC
: if and only if both “ccflag“ and ”nsfflag” are equal to “Y“, we block both customer credit card and ACH payments. If both flags are equal to “N“(or any other value), we unblock customer credit card and ACH payments. If both flags are empty/null, we make no change.

PayType
: The PayType is a biller specific payment code that gets set depending on the Payment Source and Payment Type. There are 2 methods in sending the Payment Codes: 
Standard Payment Codes
 or 
Dynamic Payment Codes
. The current Harris NorthStar DataPump Integration 
only 
supports the following payment types for either Standard Payment Codes and Dynamic Payment Codes: 
Credit/Debit Card
Cash
PayPal
Venmo
PayPal Credit
OBD
We use the following Advanced Settings for both 
Standard and Dynamic Payment Codes
 to send payment codes for the PayType depending on it’s Payment Type: 
Advanced Setting 64 - DataPump Payments CC Code
Advanced Setting 65	- DataPump Payments ACH Code
Advanced Setting 104 - DataPump Payments Cash Code
Advanced Setting 116 - DataPump Payments Paypal Code
Advanced Setting 117 - DataPump Payments Paypal Credit Code
Advanced Setting 118 - DataPump Payments Venmo Code
PayType - Standard Payment Codes
This format that we use to send payment codes relies on the amount of payment codes that are in the Advanced Setting’s value to determine which payment code we are sending in orientation to the position of the code in the Advanced Setting.  
Below is the current payment source support we offer per Advanced Setting:
Advanced Setting 64 - Web CC Code|IVR CC Code|Autopay CC|Kiosk CC Code
Advanced Setting 65	- Web ACH Code|IVR ACH Code|Autopay ACH Code|Kiosk ACH Code|HB Eck
Advanced Setting 104 - Kiosk Cash Code|Web Service Cash Code
Advanced Setting 116 - Web Paypal Code|IVR Paypal Code|Autopay Paypal Code|Kiosk Paypal Code
Advanced Setting 117 - Web Paypal Credit Code|IVR Paypal Credit Code|Autopay Paypal Credit Code|Kiosk Paypal Credit Code
Advanced Setting 118 - Web Venmo Code|IVR Venmo Code|Autopay Venmo Code|Kiosk Venmo Code
Special Note: The default value for Advanced Setting 64, 65, 116, 117, and 118 when the payment type is neither IVR, AutoPay, or Kiosk, we use the first value in the Advanced Setting. This value is usually the Web Service payment code. The only exception is for Advanced Setting 104; we use the last value of the Advanced Setting to send the payment code if the payment type is not Kiosk. 
Example 1: Using the Advanced Setting example below, if we are sending a Credit Card payment via DataPump, we are going to use Advanced Setting 64 to get the payment code we are sending for it’s payment source. If the payment’s payment source is AutoPay, we are going to use the 3rd value between the pipe-delimited value and set the PayType to AUTOCC as such.
Advanced Setting 64: WEBCC|IVRCC|AUTOCC|KIOSKCC|POSCC
Example 2: Using the Advanced Setting example below, if we get a payment with a payment source of Cash, we are going to use Advanced Setting 104. If the payment type for this payment record is also Kiosk, we are going to use the first value of the Advanced Setting.
Advanced Setting 104: Kiosk PaymentCode|Web PaymentCode
PayType - Dynamic Payment Codes
This format relies on a specific format in the Advanced Settings to send payment codes per payment source. The format to send any payment code for any payment source can be seen below: 
PaymentSourceID~Biller_Specific_Payment_Code
To add multiple payment codes, you can add more by adding a comma after each code. To create a Dynamic Payment Code, you will need to use the ID for the Payment Type that is being requested to send. Here is a list of all the Payment Type IDs we currently have (as of 5/23/2023):

Special Note: The default value for each Payment Type (Advanced Settings) can be added by typing “Default” followed by a tilde, followed by the biller-specific payment code. This value will be the default payment code if the payment source ID that a payment has that does not exist in the Advance Setting. The format can look like this: 
Advanced Setting 64: 
10~IVR CC,2~Biller Portal CC,8~POS CC,Default~Web CC
Example 1.1: If a biller wants to receive payment codes for payment sources POS (8), Pay By Text (23), and Kiosk (19), when the payment was made using a Credit/Debit Card, we can use Advanced Setting 64 with the following value where “Web CC” is Default:
Advanced Setting 64:
8~POS CC,23~Pay-By-Text CC,19~KIOSK CC,Default~Web CC
Example 1.2: Using the example in 1.1, if a payment comes in with a Payment Source ID = 23, the PayType send in the DataPump request will be “PayByTextCCCode”. 
Example 1.3: Using the example in 1.1, if a payment comes in with a payment source that does not exist amongst the Payment Codes in the Advance Setting, the PayType will be set to the Default value as  “Web CC“.
PayType - CONVERTING STANDARD PAYMENT CODES TO DYNAMIC PAYMENT CODES: 
Here are some examples of how to convert Standard Payment Codes to Dynamic Payment Codes:
NOTE: Values in brackets (including brackets) are where the payment code belongs. 
Example 1 - Converting Advanced Setting 64 (CC) from Standard to Dynamic Payment Codes: 
STANDARD PAYMENT CODES
[Insert WebCode Here]|[Insert IVRCode Here]|[Insert AutopayCode Here]|[Insert Kiosk Here]

DYNAMIC PAYMENT CODES
10~[Insert IVRCode Here],15~[Insert AutopayCode Here]|,19~[Kiosk CC],Default~[Insert WebCode Here]
Example 2 - Converting Advanced Setting 65 (ACH) from Standard to Dynamic Payment Codes: 
STANDARD PAYMENT CODES
[Web ACH]|[IVR ACH]|[Autopay ACH]|[OBD]

DYNAMIC PAYMENT CODES
10~[IVR ACH],15~[Autopay ACH],Default~[Web ACH],7~[OBD]
Example 3 - Converting Advanced Setting 104 (Cash) from Standard to Dynamic Payment Codes:  
STANDARD PAYMENT CODES
[Kiosk Cash]|[Web Service Cash]

DYNAMIC PAYMENT CODES
1~[Web Service] Cash,Default~[Kiosk Cash]

UploadGenericBillerPrintFile
http://www.invoicecloud.com/stratuswebservices/HarrisNorthstar/UploadGenericBillerPrintFile
Biller sends NorthStar file contents via web method within StratusWebService that uploads invoices to our system that can be either Water (InvoiceTypeID = 1), Electric (InvoiceTypeID = 2), or Utility (InvoiceTypeID = 42). 
Depending on what InvoiceTypeID and if they want PDFs to be setup for them using our logic, we use the following InvoiceFileTypeID to insert the invoice record into our system: 
InvoiceTypeID = 1, PDF=1 →  InvoiceFileTypeID = 99
InvoiceTypeID = 1, PDF=0 →  InvoiceFileTypeID = 100
InvoiceTypeID = 2, PDF=1 →  InvoiceFileTypeID = 97
InvoiceTypeID = 2, PDF=0 →  InvoiceFileTypeID = 98
InvoiceTypeID = 42, PDF=1 →  InvoiceFileTypeID = 325
InvoiceTypeID = 42, PDF=0 →  InvoiceFileTypeID = 325
If the web request contains:
InvoiceTypeID = 1, 2, 42 AND PDF = 1, we hardcode the InvoiceFormatGUID to 
B3D8E0C6-0A66-4566-BD1F-480C87121EA6
InvoiceTypeID = 1, 2, 42  AND PDF = 0, the biller is going to be Template and will use their own InvoiceFormatGUID that is created for this biller. 
UploadGenericBillerDailyBalanceFile
http://www.invoicecloud.com/stratuswebservices/HarrisNorthstar/UploadGenericBillerDailyBalanceFile
Biller sends NorthStar file contents via web method within StratusWebService that does an Adjustment on an Invoice.
Uses AdjustmentFileTypeID = 59 to convert the web request into an Adjustment Record.
