Determining future state - other options we talked about:
In the future, we’d like to ensure that each application gets its own instance of Redis. For example, MyIIS and CRM both share 
redis-east-appinfra-iaas-prod
. The better pattern would be to give each their own instance, i.e. 
redis-myiis-prod
 and 
redis-crm-prod 
(these are likely not the actual names we’d give newly created Redis instances). This is a common theme to any approach we take going forward. 
Another common theme to any direction we take is the adoption of clustered Redis instances, at least in production. Clustered instances allow for more flexibility and scalability of the instance while helping us to optimize cost.
Finally, each higher environment should get its own instance of Redis*. Lower environments can share. Internal apps (not customer facing) can also share. That yields a total of 10 Redis instances:

*This is the current suggestion. The question: should we have a single instance of Redis for all nonproduction environments, or do we want to give each their own? For example, should devtest, regression, and UAT all use the same instance of Redis? Or should they each have their own? Or, would we like to ensure that preview has its own, but allow the others to share? In theory, each application in each environment should get its own instance in order to best create parity between environments. This is the more expensive approach, though.
